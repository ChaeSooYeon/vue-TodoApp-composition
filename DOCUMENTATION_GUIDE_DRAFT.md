# 문서 작성 가이드

이 문서는 `todo-app-composition` 프로젝트의 문서를 일관된 기준으로 관리하기 위한 초안이다.

목표는 사람이 바뀌거나 다른 AI가 작업하더라도 비슷한 문서 구조와 표현 기준을 유지하는 것이다.

## 1. 문서별 역할

### `README.md`

- 프로젝트 소개 문서로 사용한다.
- 이 프로젝트가 무엇인지, 무엇을 학습하기 위한 프로젝트인지 먼저 설명한다.
- 현재 구현된 기능, 실행 방법, 구조 포인트, 학습 포인트를 담는다.
- 상세한 작업 히스토리나 문제 해결 과정은 넣지 않는다.

### `PROJECT_HISTORY.md`

- 사용자와 AI의 협업 흐름을 기록하는 타임라인 문서로 사용한다.
- 어떤 요청이 있었고, 어떤 변경이 이어졌는지 시간 흐름 중심으로 정리한다.
- 기능 결과를 장황하게 다시 설명하지 않는다.
- 가능하면 작업자와 날짜를 함께 남긴다.

### `TROUBLESHOOTING.md`

- 실제로 겪은 문제와 원인, 해결 방법을 정리하는 문서로 사용한다.
- 재현 가능하거나 구현 과정에서 의미 있었던 이슈만 남긴다.
- 설계 회고나 기능 소개는 넣지 않는다.

## 2. 공통 작성 원칙

- 문서마다 목적이 겹치지 않도록 역할을 분리한다.
- 같은 내용을 여러 문서에 반복해서 쓰지 않는다.
- 현재 코드 기준으로 확인 가능한 내용만 기록한다.
- 과거 제안과 현재 구현이 다르면 현재 구현 기준으로 정리한다.
- 문장은 짧고 명확하게 유지한다.
- 기능 단위로 너무 잘게 쪼개지 말고, 하나의 흐름이면 한 섹션으로 묶는다.

## 3. 문서별 포함 기준

### `README.md`에 포함할 것

- 프로젝트 소개
- 학습 목적
- 현재 구현 기능
- 구조 포인트
- 실행 방법
- 학습 포인트
- 실행 화면

### `README.md`에 넣지 않을 것

- 사용자/AI 작업 타임라인
- 세부 트러블슈팅 사례
- 커밋 단위의 변경 기록

### `PROJECT_HISTORY.md`에 포함할 것

- 날짜 또는 단계 기준 작업 흐름
- 사용자 요청
- AI가 수행한 분석, 수정, 검증, 문서화
- 기능 단위로 묶은 주요 변경 기록

### `PROJECT_HISTORY.md`에 넣지 않을 것

- 현재 기능 전체 목록의 반복
- 재현 절차 중심의 오류 해결 상세
- README 수준의 프로젝트 소개 반복

### `TROUBLESHOOTING.md`에 포함할 것

- 문제 상황
- 원인 분석
- 해결 방법
- 구현 중 배운 점이 있다면 문제 해결 맥락 안에서만 짧게 기록

### `TROUBLESHOOTING.md`에 넣지 않을 것

- 설계 방향 자체만 다루는 메모
- 전체 작업 타임라인
- 완료 기능 홍보성 설명

## 4. `PROJECT_HISTORY.md` 작성 형식

- 가능하면 날짜 기준으로 큰 구분을 만든다.
- 같은 기능 흐름은 하나의 섹션으로 묶는다.
- 각 항목은 `사용자:` / `AI:` 형식으로 역할을 구분한다.
- 세부 코드 설명보다는 작업 흐름과 의사결정이 보이도록 작성한다.

예시:

```md
### 2026-03-27

#### 일괄 삭제 기능 추가

- 사용자: 전체 항목 삭제 기능을 구현함
- AI: 변경 파일을 확인하고 빌드 검증 후 커밋함
- 사용자: 전체 삭제 전 확인창 추가를 요청함
- AI: `window.confirm`을 추가하고 다시 검증함
```

## 5. `TROUBLESHOOTING.md` 작성 형식

- 한 문제당 하나의 섹션으로 정리한다.
- 아래 순서를 기본 형식으로 사용한다.

```md
## 문제 제목

### 증상

- ...

### 원인

- ...

### 해결

- ...
```

## 6. 중복 점검 체크리스트

문서를 수정하기 전에 아래 항목을 확인한다.

- 이 내용이 이미 다른 문서에 더 적절한 형태로 들어가 있지 않은가
- 현재 구현이 아니라 과거 논의 내용을 적고 있지 않은가
- 같은 기능을 두세 개 섹션으로 과하게 쪼개고 있지 않은가
- 프로젝트 소개, 작업 기록, 문제 해결이 서로 섞여 있지 않은가
- 다른 AI가 읽어도 어느 문서에 무엇을 써야 할지 바로 판단할 수 있는가

## 7. 권장 작업 순서

- 먼저 이 내용이 어느 문서에 들어가야 하는지 판단한다.
- 기존 문서와 중복되는지 확인한다.
- 현재 코드와 맞는지 확인한다.
- 필요한 경우 초안 파일에서 먼저 정리한 뒤 기존 문서 반영 여부를 결정한다.
