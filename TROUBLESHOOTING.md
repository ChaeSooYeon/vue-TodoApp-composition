# Todo App 트러블슈팅

구현 과정에서 겪은 문제와 해결 내용을 간단히 정리한 문서입니다.

## 1. `props.todos`가 바뀌어도 목록이 바로 갱신되지 않던 문제

### 증상

- `TodoList.vue`에서 props로 받은 목록이 바뀌어도 화면이 기대처럼 다시 그려지지 않았다.

### 원인

- 부모 컴포넌트에서 `computed`와 `ref`를 섞어 `todos` 흐름이 꼬여 있었다.
- 파생된 목록을 직접 수정하는 코드가 있어 원본 상태와 화면 상태가 섞여 있었다.
- `computed` 내부에서 상태를 다시 대입하는 구조가 반응형 흐름을 복잡하게 만들었다.

### 해결

- `todos`는 `localStorage`에서 초기 1회만 읽어오도록 정리했다.
- 화면용 목록은 `computed`에서 필터링과 정렬만 하도록 단순화했다.
- 추가, 삭제, 수정, 완료 처리는 모두 `todos.value`를 직접 갱신하는 구조로 통일했다.

## 2. 수정 input을 항목별로 토글하기 어려웠던 문제

### 증상

- 수정 input을 보여주고 숨기는 처리를 하려 했지만, 어떤 항목이 수정 중인지 구분하기 어려웠다.

### 원인

- `editable` 같은 단일 boolean 상태만으로는 여러 todo 중 어느 항목이 수정 중인지 표현하기 어려웠다.

### 해결

- `editingId`를 두고 현재 수정 중인 todo의 `id`를 저장하도록 바꿨다.
- `editingId === item.id`일 때만 input을 렌더링하게 해서 항목별 제어가 가능해졌다.
- 수정 완료 후에는 `editingId`를 `null`로 돌려 수정 모드를 종료했다.

## 3. `localStorage`에 저장했는데 화면이 바로 바뀌지 않던 문제

### 증상

- 수정 후 `localStorage`에는 값이 저장되는데 화면에 즉시 반영되지 않는 것처럼 보였다.

### 원인

- `localStorage`는 Vue의 반응형 시스템이 아니기 때문에, 저장만으로는 재렌더링이 일어나지 않는다.
- 화면 갱신의 기준은 `localStorage`가 아니라 Vue의 반응형 상태여야 한다.

### 해결

- 먼저 `todos.value`를 갱신해서 Vue가 반응하도록 만들고,
- 그 다음 저장은 `setTodoLocalStorage()`로 따로 수행하도록 역할을 분리했다.

## 4. 카드 너비를 넓혔는데도 화면이 계속 좁아 보이던 문제

### 증상

- `.todo`의 `max-width`를 늘렸는데도 실제 화면에서는 카드가 여전히 좁게 보였다.

### 원인

- 바깥 래퍼인 `#app`이 전체 폭을 갖고 있지 않아, 내부 `.todo`가 넓어질 기준이 부족했다.

### 해결

- `.todo`에 `max-width`를 조정하고,
- `#app { width: 100%; display: flex; justify-content: center; }`를 추가해 카드가 의도한 최대 폭까지 확장되도록 수정했다.

## 5. 저장 유틸에서 데이터를 바로 가공해도 되는지에 대한 고민

### 고민

- `useLocalStorage.js`에서 데이터를 읽어오자마자 정렬하거나 가공하는 방식이 괜찮은지 고민이 있었다.

### 정리한 기준

- `useLocalStorage.js`는 읽기/쓰기만 담당한다.
- 필터링, 정렬, 개수 계산 같은 화면용 가공은 `App.vue`의 `computed`에서 처리한다.

### 이유

- 저장 유틸이 UI 정책까지 담당하면 재사용성이 떨어진다.
- 화면 요구사항이 바뀌더라도 저장 로직은 그대로 둘 수 있어 구조가 더 안정적이다.

## 6. `App.vue`에 상태와 핸들러가 너무 많이 모이던 문제

### 고민

- 상태, 계산값, CRUD 핸들러가 모두 `App.vue`에 모이면서 파일의 역할이 점점 커지고 있었다.

### 정리한 기준

- `App.vue`는 화면 조립에 집중한다.
- 투두 도메인 상태와 비즈니스 로직은 `useTodos.js`로 분리한다.

### 해결

- `todos`, `current`, `computedTodos`, 카운트 계산, CRUD 핸들러를 `useTodos.js`로 이동했다.
- `App.vue`는 hook의 반환값을 받아 하위 컴포넌트에 연결만 하도록 단순화했다.

## 7. 한글 입력 중 Enter로 todo가 여러 번 등록되던 문제

### 증상

- 마우스로 등록할 때는 괜찮았지만, 한글 입력기 사용 중 `Enter`로 todo를 추가하면 조합 중인 값이 여러 번 등록됐다.

### 원인

- 한글 IME 조합 중에는 입력 확정과 `Enter` 키 이벤트가 함께 발생할 수 있다.
- 조합 상태를 고려하지 않고 `keydown.enter`에서 바로 등록하면 중간 조합 문자열까지 todo로 추가될 수 있다.

### 해결

- `TodoInput.vue`에 `compositionstart`, `compositionend`를 추가해 조합 상태를 추적했다.
- `event.isComposing` 또는 조합 중 상태일 때는 `Enter` 등록을 막았다.
- 입력 단계와 `useTodos.js` 양쪽에서 빈 문자열 추가도 함께 차단했다.
